# 无人机项目维护总览

## 1. 文档目的

这份文档不是为了重构代码，而是为了把当前工程的真实结构、运行流程、模块职责和后续维护入口说清楚。

适用场景：

- 你自己过一段时间后回来看工程，能快速找回思路
- 项目交接给下一个维护者时，对方能快速理解系统
- 训练自己从“能写功能”走向“能描述系统、管理复杂度”的架构思维

## 2. 项目定位

从当前代码来看，这个工程更像是一个“无人机接收执行端”工程，而不是完整飞控算法工程。

它当前主要完成这些事情：

- 初始化 STM32F103 的 GPIO、串口、SPI、TIM
- 初始化 FreeRTOS 调度环境
- 初始化 2.4G 无线模块 SI24R1，并进入接收模式
- 接收遥控器项目发来的数据包
- 解析遥控器摇杆和按键信息
- 管理 LED 状态显示
- 初始化 4 路电机 PWM 输出
- 维持一个电源保持信号

它当前还没有真正形成完整闭环的部分：

- 没有姿态传感器采集链路
- 没有姿态解算
- 没有 PID 控制器
- 没有将遥控量映射为 4 路电机混控输出
- 没有失联保护收敛逻辑
- 没有统一的系统状态机调度中心

所以从架构角度说，这个工程当前属于：

“接收控制链路 + 基础执行框架已搭好，但飞控控制核心尚未完善”

## 4. 当前启动流程

系统启动入口在 [main.c](/d:/2026/ZIQI/fly/done/Core/Src/main.c#L51)。

主流程如下：

1. `HAL_Init()`
2. `SystemClock_Config()`
3. 初始化 GPIO、USART2、TIM1~TIM4、SPI1
4. 串口打印 `"LOVE AND LIFE!"`
5. 调用 `done_task_Init()`
6. 在 `done_task_Init()` 中初始化电机和无线模块
7. 创建 4 个 FreeRTOS 任务
8. 调用 `vTaskStartScheduler()` 启动调度器
9. `main` 的 `while(1)` 实际上不再承担业务逻辑

## 5. 任务设计与职责

当前工程创建了 4 个任务。

### 5.1 `power_task`

文件位置：

- [done_task.c](/d:/2026/ZIQI/fly/done/done_task/done_task.c#L62)

职责：

- 周期性拉低再拉高 `NRST_BAT` 引脚
- 从注释看，它的意图是维持某个电源保持/防休眠信号

行为特点：

- 1 秒一次任务周期
- 中间有一次 100ms 的低电平脉冲

维护提示：

- 如果后续硬件确认这是“电源保持”或“防低功耗休眠”控制脚，应把它的设计意图写到硬件文档里
- 这个任务目前是纯周期动作，没有跟系统状态联动

### 5.2 `led_task`

文件位置：

- [done_task.c](/d:/2026/ZIQI/fly/done/done_task/done_task.c#L77)

职责：

- 根据遥控连接状态 `rc_status` 控制顶部 LED
- 根据飞机状态 `plane_status` 控制底部 LED 闪烁或常亮逻辑

已实现的状态含义：

- `eRC_UNCONNECTED`：顶部两个 LED 关闭
- `eRC_CONNECTED`：顶部两个 LED 打开
- `eDONE_IDEL`：底部 LED 每 1000ms 翻转一次
- `eDONE_NORMAL`：底部 LED 每 100ms 翻转一次
- `eDONE_HOLD_HEIGHT`：顶部 LED 常亮
- `eDONE_FAULT`：顶部 LED 关闭

维护提示：

- 这里已经有“状态驱动显示”的雏形，很适合以后扩展为统一状态机
- 但当前 `rc_status` 和 `plane_status` 基本没有被其他任务更新，状态流还没有闭环

### 5.3 `motor_task`

文件位置：

- [done_task.c](/d:/2026/ZIQI/fly/done/done_task/done_task.c#L142)

职责：

- 上电时将 4 个电机速度初始化为 0
- 调用 `Int_Motor_SetSpeed()` 下发一次 PWM

当前现状：

- 初始化后没有根据遥控输入持续更新电机速度
- 任务主体只有空循环 + 延时

这说明：

- “电机输出框架”已经搭好
- “控制分配逻辑”还没有接入

### 5.4 `rx_task`

文件位置：

- [done_task.c](/d:/2026/ZIQI/fly/done/done_task/done_task.c#L161)

职责：

- 周期调用 `App_Data_Receive(&joystick_key_handle)`
- 从无线模块接收遥控器数据
- 将结果写入全局结构体 `joystick_key_handle`

行为特点：

- 500ms 一次轮询
- 每次打印 `"radio_task"`

维护提示：

- 如果后续要做实际飞控，这个周期偏慢，通常遥控接收更新频率要更高
- 当前任务只完成“接收+解析”，没有把接收到的数据驱动到状态机或电机控制

## 7. 遥控数据链路

数据流路径如下：

1. 遥控器项目发出 32 字节数据包
2. 无人机端 SI24R1 进入 RX 模式接收数据
3. `rx_task` 周期调用 `App_Data_Receive()`
4. `App_Data_Receive()` 调用 `SI24R1_RxPacket(rxbufer)`
5. 若收到数据，先检查协议 ID
6. 再做校验和检查
7. 解析为 `pitch/roll/yaw/throttle/shutdown/hold_height`
8. 写入 `joystick_key_handle`
9. 串口打印解析结果

协议关键点在：

- [App_Data.h](/d:/2026/ZIQI/fly/done/App/Inc/App_Data.h#L7) 中定义了 3 个协议标识字节
- [App_Data.c](/d:/2026/ZIQI/fly/done/App/Src/App_Data.c#L11) 中进行了协议头检查
- [App_Data.c](/d:/2026/ZIQI/fly/done/App/Src/App_Data.c#L15) 中进行了前 13 字节和校验

从当前代码推断，数据格式大致如下：

| 字节范围 | 含义 |
| --- | --- |
| `0~1` | `pitch` |
| `2~3` | `roll` |
| `4~5` | `yaw` |
| `6~7` | `throttle` |
| `8` | `shutdown` |
| `9` | `hold_height` |
| `10~12` | 协议标识 |
| `13~16` | 校验和 |

注意：

- 实际发送端必须和这里完全一致，否则解析会失败
- 这个工程和“遥控器项目”之间最关键的耦合点就是这个协议定义

## 10. 当前工程的维护风险点

这部分不是批评，而是交接时最需要提前提醒下一个维护者的地方。

### 10.1 `led_task` 的状态来源不完整

LED 行为依赖：

- `rc_status`
- `plane_status`

但当前工程里几乎没有看到这些状态被动态更新的逻辑，所以 LED 只能反映初始状态或局部逻辑。

### 10.2 `motor_task` 还没有接入控制闭环

当前只是初始化为 0，没有使用遥控输入去更新 4 路电机输出。

### 10.3 `rx_task` 接收周期偏慢

500ms 的接收周期更适合调试，不太适合飞行控制输入。

### 10.4 模块之间依赖以全局变量为主

比如：

- `joystick_key_handle`
- `rc_status`
- `plane_status`

这在小项目里很常见，但随着功能增长，容易出现“谁在改状态、谁在消费状态”越来越难追踪。

### 10.5 协议文档目前隐含在代码里

遥控器和无人机端的通信协议主要靠代码约定，没有独立协议文档。后续只要另一端工程改了字节顺序，这边就可能失配。

### 10.6 部分驱动语义需要再次确认

例如 [Int_Led.c](/d:/2026/ZIQI/fly/done/Internet/Src/Int_Led.c#L14) 中 `Int_LED_on()` 和 `Int_LED_off()` 当前都写成了 `GPIO_PIN_SET`，这在阅读层面会让维护者困惑。即使硬件是低电平点亮，也建议文档里明确“有效电平”。

## 12. 推荐的架构化演进方向

这部分不是要求你现在改代码，而是帮你建立“下一步应该往什么方向长”的系统感。

建议未来逐步补齐下面 5 个中心模块：

### 12.1 输入中心

职责：

- 统一接收遥控器数据
- 做丢包检测
- 做失联判定
- 输出标准化控制命令

### 12.2 状态中心

职责：

- 统一维护 `IDLE / ARMED / NORMAL / HOLD / FAULT / LOST_LINK`
- 明确每个状态允许哪些操作

### 12.3 控制中心

职责：

- 把遥控输入、姿态反馈、状态机结果组合起来
- 输出 4 路电机目标转速

### 12.4 执行中心

职责：

- 只负责把目标电机值写入 PWM
- 不掺杂业务判断

### 12.5 监控中心

职责：

- 串口打印
- 故障码输出
- LED/蜂鸣器提示

当一个项目形成这几个“中心”之后，维护复杂度会明显下降。
