# C语言面试题统一整理模板

## 1. 文档目的

这份模板不是单纯帮你“记题目”，而是帮你把面试题整理成一套长期可复用的知识系统。

适合你的原因：

- 你现在是 MCU / 嵌入式方向
- 后面不仅要准备面试，还要把知识真正内化成工程能力
- 你需要的不是“抄一堆题”，而是“建立自己的知识卡片库”

所以这份模板的目标是：

1. 帮你统一整理格式
2. 帮你搭出题库结构
3. 帮你区分“知识点”“高频题”“深度追问”
4. 帮你以后复习时能快速筛选

---

## 2. 推荐的整理方式

我建议你不要只按：

- 高频
- 中频
- 低频
- 深度

来做主目录。

更好的方式是：

### 主目录按知识模块分类

比如：

- 数据类型
- 指针
- 内存
- 关键字
- 编译链接
- 位操作
- 嵌入式 C

### 每道题再打标签

比如：

- 频率：高频 / 中频 / 低频
- 难度：基础 / 中等 / 深度
- 方向：通用 C / 嵌入式 C / MCU / RTOS
- 状态：未掌握 / 已理解 / 可讲清 / 可手写

这样整理的好处是：

1. 平时学习时按知识体系补
2. 面试冲刺时按“高频 + 未掌握”筛选
3. 长期积累时不会变成一堆零散题目

## 4. 每个字段怎么写

### 4.1 题目

直接写面试官会问出来的话。

例如：

- `volatile 有什么作用？`
- `指针和数组有什么区别？`
- `malloc 和 calloc 的区别是什么？`

不要写得太像目录名。

错误写法：

- `volatile`

更推荐：

- `volatile 在嵌入式里为什么很重要？`

### 4.2 分类

写知识模块，不写频率。

例如：

- `关键字`
- `指针与内存`
- `编译与链接`
- `嵌入式 C`

### 4.3 频率

只写 3 档就够了：

- `高频`
- `中频`
- `低频`

### 4.4 难度

建议统一成：

- `基础`
- `中等`
- `深度`

### 4.5 方向

建议按你自己的发展方向打标签：

- `通用C`
- `嵌入式C`
- `MCU`
- `RTOS`

### 4.6 状态

这个很重要，它能帮助你筛题复习。

建议你用这 4 档：

- `未掌握`
- `已理解`
- `可讲清`
- `可手写`

解释一下：

- `未掌握`：看过但答不出来
- `已理解`：自己看懂了
- `可讲清`：能口头讲出来
- `可手写`：能白板写代码或现场手写

### 4.7 考察点

写面试官真正想考什么。

比如题目是：

- `const 和 define 的区别是什么？`

考察点可以写：

- 预处理阶段
- 类型检查
- 作用域
- 调试可读性

## 5. 模板示例

下面给你一个完整示例。

## 题目：`volatile` 有什么作用？

- 分类：关键字
- 频率：高频
- 难度：中等
- 方向：嵌入式C / MCU
- 状态：未掌握
- 考察点：编译器优化、寄存器、中断共享变量、内存可见性

### 标准回答

`volatile` 用来告诉编译器，这个变量的值可能在程序当前执行流之外被修改，所以每次访问都要重新从内存或寄存器地址读取，不能随意优化成寄存器缓存值。

### 通俗理解

你可以把它理解成：

“这个变量很活跃，外面有人可能随时改它，你每次都要重新看，不要偷懒记住旧值。”

### 面试官想听什么

面试官通常不是只想听一句“防止编译器优化”，而是想听你知不知道它在什么场景必须用。

### 易错点

`volatile` 不能保证原子性，也不能保证线程安全，它只是限制编译器优化。

### 常见追问

1. `volatile` 和 `const` 能不能一起用？
2. `volatile` 能解决多线程竞争吗？
3. 中断共享变量为什么要加 `volatile`？

### 代码示例

```c
volatile uint8_t rx_flag = 0;

void USART_IRQHandler(void)
{
    rx_flag = 1;
}

int main(void)
{
    while (1)
    {
        if (rx_flag)
        {
            rx_flag = 0;
            // 处理串口接收事件
        }
    }
}
```

### 嵌入式/MCU场景

- 中断里修改、主循环里读取的变量
- 硬件寄存器映射变量
- DMA 改写的内存区域

### 一句话记忆

`volatile` 不是线程安全，它只是告诉编译器“每次都重新读”。

## 7. 适合你的标签体系

因为你是 MCU 工程师，所以建议你每道题额外打两个标签：

### 工程标签

- `裸机`
- `中断`
- `驱动`
- `协议`
- `RTOS`

### 面试状态标签

- `只会背`
- `能讲例子`
- `能写代码`
- `能结合项目`

这个很关键。

因为真正面试里，最加分的不是“标准答案背得对”，而是：

“我在项目里就是这么用的”

比如：

- `volatile`
- `环形缓冲区`
- `位操作`
- `状态机`

这些你都可以结合你自己的项目讲。

## 11. 一句话结论

你原来用：

- 高频
- 中频
- 低频
- 深度

这个思路本身没有错，但更适合作为“标签”，不适合作为“主目录”。

更好的整理方式是：

**按知识模块建主目录，按频率、难度、方向、状态打标签，再用统一模板记录每道题。**

这样你这份面试题库，后面不只是能帮你面试，还能慢慢变成你自己的 C 语言知识资产。
